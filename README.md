# wc — клон утилиты Unix `wc`

[![CI](https://github.com/bl4ckd4y/wc-clone/actions/workflows/ci.yml/badge.svg)](https://github.com/bl4ckd4y/wc-clone/actions/workflows/ci.yml)

Реализация утилиты `wc` (word count) на языке C. Считает в файле:

- `-l` — количество строк
- `-w` — количество слов
- `-c` — количество байт

Если флаги не указаны, выводятся все три значения. Если передано несколько файлов, в конце выводится строка `total` с суммой.

## Сборка

```bash
make
```

Собирается через GCC со следующими флагами:

```
-Wall -Wextra -Werror -std=c11 -O2
```

`-Werror` превращает любое предупреждение в ошибку компиляции — это значит, что код не соберётся, пока не будут устранены все warning'и.

## Использование

```bash
./wc file.txt          # строки, слова, байты
./wc -l file.txt       # только строки
./wc -w file.txt       # только слова
./wc -c file.txt       # только байты
./wc -lw file.txt      # строки и слова (флаги можно объединять)
./wc a.txt b.txt       # несколько файлов + строка total
```

Коды возврата:

- `0` — успех
- `1` — ошибка (неизвестный флаг, не удалось открыть файл и т.д.)

## Тесты

```bash
make test
```

Тесты написаны на bash (`tests/run_tests.sh`) и сравнивают вывод этой реализации с системным `wc`, который считается эталоном. Покрытие:

| № | Что проверяется |
|---|---|
| 1 | `-l` — количество строк |
| 2 | `-c` — количество байт |
| 3 | `-w` — количество слов |
| 4 | Вывод без флагов (все три числа) |
| 5 | Два файла — должна быть строка `total` в конце |
| 6 | Объединённые флаги `-lw` |
| 7 | Неизвестный флаг → exit code 1 |
| 8 | Несуществующий файл → exit code 1 |
| 9 | Пустой файл — 0 строк, 0 слов, 0 байт |

## Проверка санитайзерами

Для поиска утечек памяти, выходов за границы буферов и undefined behavior сборка также доступна с AddressSanitizer и UndefinedBehaviorSanitizer:

```bash
make test-asan
```

Это пересоберёт бинарник с флагами:

```
-O1 -g -fsanitize=address,undefined -fno-omit-frame-pointer
```

и прогонит на нём все тесты. Если где-то в коде есть обращение к освобождённой памяти, неинициализированному чтению или, например, знаковому переполнению — тесты упадут с подробным отчётом санитайзера.

## CI

На каждый push и pull request в ветку `main` GitHub Actions автоматически запускает пайплайн:

1. Сборка (`make`)
2. Прогон тестов (`make test`)
3. Прогон тестов под санитайзерами (`make test-asan`)

Конфигурация — в `.github/workflows/ci.yml`.

## Архитектура `src/wc.c`

Программа разбита на маленькие функции с одной ответственностью:

| Функция | Что делает |
|---|---|
| `options_init`, `counts_init` | Инициализируют структуры значениями по умолчанию |
| `parse_flags` | Разбирает аргументы командной строки и заполняет `options_t`. Возвращает `-1` при неизвестном флаге |
| `count_file` | Считает строки/слова/байты в одном открытом файле через посимвольный проход |
| `process_one_file` | Открывает файл, вызывает `count_file`, печатает результат, обрабатывает ошибки |
| `add_counts` | Прибавляет счётчики одного файла к итогу |
| `print_counts` | Печатает счётчики с учётом активных флагов |
| `main` | Связывает всё вместе: парсинг флагов, цикл по файлам, вывод `total` |

Подсчёт слов реализован через **конечный автомат** на двух состояниях (`in_word = 0/1`): при встрече пробельного символа состояние сбрасывается, при встрече непробельного — если сейчас не в слове, увеличиваем счётчик слов и переходим в состояние «в слове». Так корректно обрабатываются любые комбинации пробельных символов между словами.

## Что я узнал, делая этот проект

- Понял, как использовать конечный автомат на примере подсчёта слов: держишь одну переменную `in_word` (0 или 1), на каждом символе решаешь «остаться в том же состоянии или перейти в другое».
- Разбил монолитную функцию `main` на несколько маленьких (`parse_flags`, `process_one_file`, `add_counts`, `print_counts`), каждая делает одну вещь.
- Настроил GitHub Actions CI, который на каждый push автоматически собирает проект и прогоняет тесты, включая проверку под санитайзерами.
- Разобрался, как программа получает аргументы командной строки. `argv` — это массив строк, и чтобы обработать флаги типа `-lw`, нужны два вложенных цикла: внешний по аргументам, внутренний по символам одного аргумента.
